# 読書記録（リスト・感想）アプリ 設計方針書

## 📋 概要

[要件定義書（specification.md）](specification.md) に基づき、**クライアント完結の読書管理 UI**を実装するうえでの**アーキテクチャ・状態・永続化・画面遷移**の方針を定める。  
本リポジトリは学習用で既存に **Vite + React 19 + TypeScript**（メモアプリ）が存在するため、**同スタックの延長**として一貫性を保つ。具体的な関数名・コンポーネント名のコード例は本書に書かず、実装者の裁量の範囲でよい。

## 🔍 要件の分析

実装に直結する要点は次のとおり。

- **本の CRUD** と**状態（未読 / 読書中 / 読了）**、**日付2種（任意）**、**感想テキスト**の編集
- **一覧フィルタ**（状態別）と**永続化**（再読込後も残る）
- **レスポンシブ**と**アクセシビリティ基本**（ラベル、`button type="button"` 等）
- 対象は**紙本・電子籍のみ**、認証なし
- 要件上まだ**未確定**だった UI 挙動（詳細の開き方、削除確認、プレーン/Markdown、ソート等）に対し、**v1 としての仮定**を本書で決め、工程を前に進められるようにする

## 🛠 実装方針

### アーキテクチャ選択

- **形態**: ブラウザ上の **SPA**（**Vite** ビルド、**React** のみ。現状 `package.json` に合わせる）
- **既存アプリとの関係**:
  - **案**: 同一リポジトリ内に **「読書」機能**を**ディレクトリ単位で分離**（例: 機能（feature）用フォルダ、または `src/reading` 等）。メモ用コードと**共有するのは**「フック＋`localStorage` デバウンス保存」などの**パターン**にとどめ、**ドメイン型とストレージキーは分離**する
  - **ルーティング**: **v1 は `react-router` を導入しない**（学習用・依存を増やさない）。画面は **App 近傍の状態**で「一覧」「詳細（編集）」を切り替え、既存メモアプリの **窄幅/広幅**に近い**パネル切替**で実現する。URL を本ごとに持たないトレードオフを許容
  - 将来、URL 共有が必要なら **React Router 追加**を「拡張パス」として記載

### コンポーネント設計

- **責務の分離**（目安）:
  - **レイアウト**: ヘッダー、アプリ枠
  - **一覧**: フィルタ用の状態切替、本の行リスト
  - **行**: タイトル・状態バッジ・日付/感想の短い要約、行クリックで「選択」
  - **詳細/フォーム**: 選択中の本の全フィールド編集、削除、戻る
  - **空状態・読み込み/保存エラー**の表示（`localStorage` 失敗時のメッセージ）を 1 か所の方針で扱う
- **1 冊 1 レコード**のドメイン型（`type` 定義）を **`types` 層**に置き、UI はその型に依存させる

### データフロー

- **単一の情報源**: クライアント上は **本の配列**（例: `Book[]`）を **React 状態**として保持
- **永続化**: 既存 `useMemos` と同様、**`localStorage` + JSON シリアライズ**。**キー名はメモ用と衝突しない**別プレフィックス（例: `reading-journal:books`）を必ず用いる
- **更新タイミング**: 配列の変更のたびに**短いデバウンス**（100〜200ms 程度）のうち **1 回**保存し、再描画のたびの連続 `setItem` を避ける
- **読み込み**: 起動時に 1 回だけ読み、**JSON 壊損**時は**空配列**に戻し、**ユーザー向け短い警告**（コンソールまたは軽いバナー）に留める
- **日付**: フォームは **`input type="date"`** を想定。保存形式は **日付のみの文字列**（**`YYYY-MM-DD`**）を推奨。未選択は `null` / 空文字のどちらかに**統一**（実装で一貫すれば可）
- **並び**: **更新日時（`updatedAt`）降順**を既定（下記「決定」）。表示用にソートは**非破壊的**（元配列のコピーを並べ替えて表示）でも、保存配列内をソートし直すでも可

## 🔄 実装方法の選択肢と決定

### 選択肢 1: 詳細表示 — 専用ルート（React Router）

- 説明: `/books/:id` などにして URL と同期
- 評価: 深いリンク・ブックマークに有利。学習用リポに **依存と設定が増える**、v1 スコープに対してやや重い

### 選択肢 2: 詳細表示 — 単一 SPA 内の状態遷移（採用）

- 説明: **「選択中の本 ID」**（または行オブジェクト）を状態で持ち、同一ツリー内で**一覧/詳細**を出し分け。既存メモ App の**窄幅で一覧/編集の切替**と同系統の UX にできる
- 評価: **追加パッケージ不要**、要件の「一覧＋編集」に素直。URL に履歴が乗らない**欠点**は v1 では許容

**決定: 選択肢 2**  
**理由**: 学習用・既存構成との整合。要件未確定の「スライドオーバ」より**実装の見通しが立てやすい**。窄幅は**一覧戻る**＋**詳細**の 2 面で足りる想定

### 選択肢 3: 永続化 — `indexedDB` / `Dexie` 等

- 説明: 大容量・インデックス向け
- 評価: 本の件数は想定上オーダー **10²〜10³ 程度**で足りるケースが多く、**要件に必須でない**

### 選択肢 4: 永続化 — `localStorage` + デバウンス（採用）

- 説明: 要件・既存パターンと一致
- 評価: **十分**、実装負担が小さい

**決定: 選択肢 4**

### その他 v1 の仮定（候補と決定）

| 論点 | 候補 | v1 決定 | 理由 |
|------|------|---------|------|
| 感想 | プレーン / Markdown | **プレーン**（`textarea`） | 要件未確定のため、**拡張しやすい単純形** |
| 削除 | 即時 / 確認 | **ブラウザ `confirm` で 1 回確認** | 依拠少・仕様の「未解消」を**実用レベルで前進** |
| 既定の並び | 登録/更新/読了日 | **`updatedAt` 新しい順** | 「最近触った本」に近く、行動的に扱いやすい |
| `startedOn` 必須化 | 状態連動 / 常に任意 | **常に任意** | 仕様上「読了日必須でない」に揃え、**バリデーションを単純**に保つ |
| 紙/電子 | 列なし / 列あり | **v1 は列を持たない** | 要件の「分類用フィールド未解決」に合わせ、**YAGNI** |

**決定内容は要件定義の「合意」に近い扱いのため、specification 側の整理を推奨**（下記「要件との整合」参照）。

## 📊 技術的制約と考慮事項

- **TypeScript**: 本ドメインの型は **`type` キーワード**で定義（プロジェクトルール）
- **`localStorage` 容量**: 極端に長い感想で溢れる可能性はあるが、v1 の想定件数なら**実務上低リスク**（将来はエクスポート等）
- **日付のタイムゾーン**: `YYYY-MM-DD` の**日付部のみ**に寄せ、**日付の解釈をローカル日付**に揃える（`Date` オブ深夜の誤差に注意し、**表示・比較のルール**を一貫）
- **アクセシビリティ**: フィルタは **tab で辿れる**、フォームに **紐づく `label`**
- **~~メモアプリとの共存~~**（**合意**）: **メモ用コードは読書アプリで上書き・削除してよい。** 同一リポ内の二重 UI は**作らない**。旧メモ用 `localStorage` キーは**残っても可**（使われなくなる）が、**読書専用キー**に一本化する
