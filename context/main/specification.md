# 読書記録（リスト・感想）アプリ 要件定義書

## 📋 概要

ユーザーが**紙の本と電子書籍（以下「本」）**を登録し、**一覧で把握**しながら、**読了状況**と**自由記述の感想・メモ**、および**読み始めた日・読み終えた日**を残せる Web アプリ。  
Web 記事・雑誌切り抜き等は **v1 の対象外**（本に限定する）。学習用リポジトリ内での実装を想定し、**認証なし・データは主にブラウザ内**で完結する（保存方式は下記制約と未解決の質問に委ねる）。

## ✅ 合意済み（プロダクトの前提）

- **扱うもの**: **紙の本と電子書籍のみ**（それ以外のメディアは扱わない）
- **v1 で扱う情報**: **タイトル**、**状態**、**感想**、**本を読み始めた日**、**読み終えた日（読了日）**
- **日付のルール（合意）**: **読了日は必須にしない**（空でもよい。状態が読了でも日付未入力可）

## 🎯 要求

- 本を 1 件ずつ**登録**でき、**一覧**に表示できる
- 各本に**タイトル**を付けられる（必須）
- **感想（またはメモ）**をテキストで登録・更新できる
- **本を読み始めた日**および**読み終えた日**を、日付として登録・更新できる
- 登録した本の**追加・表示・編集・削除**ができる
- 一覧が増えたとき、**未読 / 読書中 / 読了**の**状態**で区別し、**状態で絞り込み**ができる
- **タイトルに対する検索**（**部分一致**でよい。2026-04-23 合意）で一覧を**さらに絞れる**こと。**タグ**は**v1 では扱わない**（v2 候補）
- ページ再読み込み後も**データが消えない**こと
- 主要操作が**スマートフォン幅**でも行える**レスポンシブ**であること
- ボタンに適切な `type`、フォームにラベル、操作がキーボードで辿りやすい**アクセシビリティの基本**に配慮すること

## 📝 機能仕様

### 画面/コンポーネント構成

- **アプリ名／ヘッダー**: 本の読書記録であることが分かるタイトル
- **新規登録**: 必須項目を満たしたうえで「登録（または保存）」できる入力エリア
- **一覧**:
  - 本をカード行または行リストで列挙
  - 各項目に、**タイトル**、**状態**、**感想の有無や冒頭**のサマリ（必要なら**読了日**など日付の一部）
  - 行クリック等で**詳細（編集）**に遷移。設計方針では **専用 URL（ルーター）は v1 では使わず**、**同一 SPA 内の画面切替**（広幅は一覧＋パネル、狭幅は一覧⇔編集切替可）を原則とする（[design-doc.md](design-doc.md)）
- **詳細・編集**:
  - タイトル、状態、感想、**読み始めた日**、**読了日**の編集
  - 削除操作: **v1 ではブラウザの `confirm` で 1 回確認**する（[design-doc.md](design-doc.md)）
- **フィルタ**: 一覧上で、状態（すべて / 未読 / 読書中 / 読了）の切替。併せて**タイトル検索**用の文字列入力（**クリア**で戻る）
- **空状態**: 登録 0 件のとき、登録方法を促す短い案内
- **エラーメッセージ・無効化**: 必須未入力のときは登録不可（ボタン無効化 or エラー文のいずれか）

### 振る舞い

- **登録**: タイトル空は不可。初回登録時、状態のデフォルト（例: 未読）をルール化する
- **編集**: 既存行のタイトル・状態・感想・**各日付**を上書き保存できる
- **削除**: **v1** は**確認付き**（`confirm`）。即時削除のワンクリックは採用しない
- **状態**:
  - 3 区分: **未読** / **読書中** / **読了**（ラベルは日本語で統一。英語 key は実装定義可）
  - 状態の変更は一覧または詳細のどちらか一方以上で行えること
- **日付**:
  - **読み始めた日**: 日単位（**年月日**。**時刻は扱わない**想定）。**v1 では常に未入力可**（必須にしない）（[design-doc.md](design-doc.md)）
  - **読了日（読み終えた日）**: 同様に日単位。**必須としない**（合意）。未記入のまま状態だけ「読了」にしてもよい
- **感想**:
  - **v1 はプレーンテキスト**（`textarea`）。Markdown は v2 以降で検討
  - 長文は詳細で編集。一覧では全文表示しない

### 制約条件

- **認証・マルチユーザー**は v1 では対象外
- 対象は**本（紙・電子）**に限定。URL 主の記事管理は**やらない**
- **外部 API による書誌の自動補完**（ISBN 検索等）は**必須にしない**
- **オフライン同時編集**の衝突解消は求めない
- 学習用のため、**技術スタック**は本書では固定せず、リポジトリ方針に従う

## 📊 データ要件

### 本（1 冊）に保持する属性（v1・合意版）

| 属性 | 型のイメージ | 必須 | 説明 |
|------|----------------|------|------|
| `id` | 文字列 | 必須 | 一意識別子 |
| `title` | 文字列 | 必須 | 本のタイトル。空不可 |
| `status` | 列挙 | 必須 | 未読 / 読書中 / 読了 いずれか |
| `notes` または `review` | 文字列 | 任意 | 感想・メモ。空でもよい |
| `startedOn` | 日付 | 任意 | **読み始めた日**（年月日。ISO 日付文字列等、実装定義可） |
| `finishedOn` | 日付 | 任意 | **読了日**（年月日。同上）。**必須としない**（合意） |
| `createdAt` | 数値（時刻） | 推奨 | このアプリ上の登録日時（監査用） |
| `updatedAt` | 数値（時刻） | 推奨 | 最終更新 |

- **並び順のデフォルト**: **`updatedAt`（最終更新）の新しい順**（[design-doc.md](design-doc.md)）
- **永続化**: JSON 相当。保存先は実装定義。破損時はフォールバック＋警告を推奨

## 🔄 インタラクション

- **クライアント完結**を前提: 利用者の端末上にデータが保存される
- 将来、同期やバックエンドを足す場合は、**表示・永続**の差し替えしやすい責務分離を推奨（必須ではない）

## ❓ 未解決の質問

（**2026-04-23 更新**）以下は、当初の論点に対する**合意**の記録。v1 実装でブレる箇所は、上の **要求 / 機能仕様** も併せて参照する。

1. **スライドオーバ等の高度な UI**（**合意**）: **v1 では採用しない**。[design-doc.md](design-doc.md) の一覧＋パネル / 狭幅切替のまま
2. **JSON エクスポート／インポート**（**合意**）: **当面不要**。予定なし扱いでよい
3. **検索・タグ**（**合意**）: **v1 はタイトル検索（部分一致）のみ。タグは v2 候補**（要求に反映済み）
4. **紙 / 電子**（**合意**）: **データ列では区別しない**（必要ならタイトル表記の運用に任せる）
5. **体験拡張の時期**（**合意**）: **React Router** を **v1.1 前後**に**優先**して検討。**Markdown 感想**・**ソート基準の追加**は **v2 以降**で優先度を決める
